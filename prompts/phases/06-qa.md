# Phase 06 — QA

## Purpose

Validate all acceptance criteria from `proposal.md` after all tasks are implemented and reviewed. Produce a structured bug report. Do not re-architect.

## When to use

- After all tasks in the change are APPROVED (all checkboxes in `tasks.md` are done)
- In a clean session — no implementation context

## Recommended model

**Sonnet.** QA requires careful reading of acceptance criteria and precise categorization of findings.

## Token risk level

**Medium.** Reads proposal + implemented files. Risk is bounded if the file list is explicit.

## Required inputs

- `{{CHANGE_NAME}}` — the change directory name
- `{{IMPLEMENTED_FILES}}` — complete list of all files created or modified across all tasks
- `{{TEST_COMMAND}}` — test suite command (e.g. `npm test`, `pytest`)
- `{{APP_START_COMMAND}}` — how to start the application if manual/e2e testing is needed (or "N/A")

## Prompt template

```xml
<prompt>
  <role>
    You are a QA engineer. Your job is to validate acceptance criteria and produce a bug report. You do not redesign features. You do not modify tests. You report what works and what does not.
  </role>

  <task>
    Validate the implementation of change "{{CHANGE_NAME}}" against its acceptance criteria.

    1. Read proposal.md — focus on the Acceptance Criteria section.
    2. Run {{TEST_COMMAND}}.
    3. For each acceptance criterion: determine PASS or FAIL.
    4. Write qa-report.md with findings.
  </task>

  <file_allowlist>
    read:
      - openspec/changes/{{CHANGE_NAME}}/proposal.md
      - openspec/changes/{{CHANGE_NAME}}/tasks.md
      - {{IMPLEMENTED_FILES}}
    write:
      - openspec/changes/{{CHANGE_NAME}}/qa-report.md
    forbidden:
      - memory/
      - docs/training/path/
      - .claude/skills/
      - design.md (not needed for QA)
      - Any file not in the implemented list
  </file_allowlist>

  <qa_report_format>
    ## QA Report — {{CHANGE_NAME}}
    Date: {{DATE}}

    ## Test Run
    Command: {{TEST_COMMAND}}
    Result: [PASS / FAIL — N passed, M failed]

    ## Acceptance Criteria Validation
    | # | Criterion | Result | Notes |
    |---|-----------|--------|-------|
    | 1 | ... | PASS/FAIL | ... |

    ## Bug Report
    | ID | Severity | Description | File | Steps to Reproduce |
    |----|----------|-------------|------|--------------------|
    (critical / major / minor / cosmetic)

    ## Summary
    - Criteria: X/Y passed
    - Critical bugs: N
    - Major bugs: N
    - Minor bugs: N
    - Cosmetic bugs: N
    - Recommendation: [SHIP / FIX CRITICALS / FIX MAJORS THEN SHIP]
  </qa_report_format>

  <bug_severity_guide>
    critical: feature does not work at all
    major: a core user story fails
    minor: feature works but with a defect
    cosmetic: visual/UX issue, no functional impact
  </bug_severity_guide>

  <output_policy>
    - Write qa-report.md. Do not print it in full.
    - Print only the Summary section.
    - Do not fix bugs. Do not modify source files.
    - Do not re-architect. If a critical bug reveals a design flaw, flag it for human review.
  </output_policy>
</prompt>
```

## Anti-patterns to avoid

- Fixing bugs during QA (that's the Bug Fix phase)
- Re-designing features based on QA findings (go back to the spec)
- Running QA in the same session as the last task implementation
- Marking criteria as PASS without actually verifying them

## Output policy

`qa-report.md` written silently. Summary section printed only. No source file modifications.

## Next phase

If no critical/major bugs → **Phase 08 — Summary Before Clear**, then `/opsx:archive`.
If critical/major bugs → Bug Fix session (read `qa-report.md`, fix one bug per session).
If design flaw found → return to **Phase 03 — OpenSpec Planning** to update the spec.
