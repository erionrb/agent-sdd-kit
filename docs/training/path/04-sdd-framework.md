# 04 — The Spec Driven Development Framework

SDD is not a new concept — it predates AI. But it becomes essential with AI because LLMs have no persistent memory. The spec IS the memory. Without it, every agent session starts from scratch and produces inconsistent results.

---

## Why SDD matters for AI-driven development

Without SDD:
- Each agent session is a blank slate. The AI makes its own architecture decisions.
- Two developers working on the same project get different patterns, naming conventions, and code structure.
- When something goes wrong, there's no spec to compare against — you can't tell if the AI drifted or you changed your mind.
- Onboarding a new developer means explaining all the implicit decisions verbally.

With SDD:
- Every agent session reads the spec files at the start. Decisions are written down.
- The AI's output is deterministic relative to the spec, not relative to the model's training biases.
- Any team member can run the same agent and get consistent code.
- The spec is the documentation. It's always up to date because it drives the code.

A specialist creates the spec once. Any team member using that spec with the AI develops in a very similar way. Onboarding time drops dramatically.

## The 5 phases

```
+------------------+
|   1. CONTEXT     |  Who has the problem? What's the domain?
|                  |  CLAUDE.md, architecture doc, thinking model
+------------------+
         |
         v
+------------------+
|  2. SPECIFICATION|  PRD: what & why
|                  |  No architecture here. Business only.
+------------------+
         |
         v
+------------------+
|   3. PLANNING    |  Tech Pack (how) + Tasks (steps)
|                  |  Breaks the spec into executable chunks
+------------------+
         |
         v
+------------------+
|  4. EXECUTION    |  Agent reads PRD + Tech Pack + current task
|                  |  Implements, runs tests, marks task done
+------------------+
         |
         v
+------------------+
|  5. VALIDATION   |  Task Reviewer + QA (Playwright) + Bug Fix
|                  |  Separate agents, clean context
+------------------+
```

The spec (PRD + Tech Pack + Tasks) spans phases 2 and 3. It's written once and read many times. Each downstream agent reads only what it needs.

---

## The 3 core artifacts

### PRD — Product Requirements Document
**Answers: what and why.**

Written before any technical decisions. Describes the business problem, user stories, functional requirements, acceptance criteria. Zero architecture. Zero implementation detail.

Lives at: `openspec/changes/<name>/proposal.md` (in this repo, using OpenSpec)

### Tech Pack / Design
**Answers: how.**

Written by a developer (or tech lead) after reviewing the PRD. Describes the architecture, tech stack decisions, component breakdown, data flows, integration points, testing approach. References the PRD. Does not re-state business requirements.

Lives at: `openspec/changes/<name>/design.md`

### Tasks
**Answers: in what order, with what tests.**

A numbered list of atomic, executable steps. Each task is a deliverable — working code + tests. Tasks have explicit dependencies (Task 2 requires Task 1 complete). Each task has acceptance criteria.

Lives at: `openspec/changes/<name>/tasks.md`

## How this maps to the OpenSpec skills in this repo

The OpenSpec CLI skills (`.claude/skills/openspec-*/`) implement a version of SDD:

| SDD artifact | OpenSpec artifact | Command |
|---|---|---|
| PRD | `proposal.md` | `/opsx:propose` (generates all three) |
| Tech Pack | `design.md` | Generated by `/opsx:propose` |
| Tasks | `tasks.md` | Generated by `/opsx:propose` |
| Execution | Task implementation | `/opsx:apply` |
| Archive | — | `/opsx:archive` |

OpenSpec merges PRD + Tech Pack + Tasks into a single `/opsx:propose` flow. This is the **fast flow** — good for solo work and small features.

There is also a **slow interactive flow** with separate skills:
1. PRD generator skill — interactive, asks clarifying questions, stores PRD
2. Tech Pack generator skill — reads PRD, uses Context7/web search, asks technical questions
3. Task generator skill — reads PRD + Tech Pack, proposes task list for approval before generating

The slow flow produces higher quality artifacts because each phase has dedicated, focused generation with full human input.

---

## When to use which flow

### Fast flow: `/opsx:propose`
Use when:
- Solo project or you're the only one affected
- Small features (< 1 day of work)
- You already know the architecture and don't need to think through it
- Prototyping or exploration

### Slow interactive flow (separate PRD/TechPack/Task skills)
Use when:
- Multiple developers will execute from the same spec
- The feature has ambiguous requirements that need clarification
- You want a PO to own the PRD separately from the developer's Tech Pack
- The feature is large enough that task ordering and dependencies matter a lot
- You want Context7/web search to inform library and API decisions in the Tech Pack

For this repo right now, `/opsx:propose` is the available option. The separate skills need to be built (see step 08).

## The review cycle

After each task is implemented:

```
Task Execution
     |
     v
Task Reviewer (separate agent, clean context)
     |
     +-- Approved? -> Next task
     |
     +-- Approved with minors? -> Apply minors, then next task
     |
     +-- Rejected? -> Fix issues, re-run task
     
After all tasks:

QA Agent (Playwright)
     |
     +-- Generates bug report
     |
     v
Bug Fix Agent
     |
     +-- Fixes cosmetics and minors
     +-- Updates bug report status to "fixed"
```

The reviewer and QA agents always run in separate context windows — clean slates with no bias from the implementation session.

## Practical exercise

1. Run `/opsx:propose` on a small feature idea for this repo.
2. After the artifacts are generated, open `openspec/changes/<name>/proposal.md`, `design.md`, and `tasks.md`.
3. Read them in order. Ask: do the tasks flow naturally from the design? Does the design answer all the questions raised in the proposal?
4. If yes: that's a healthy spec chain. If no: identify the gaps — those are exactly what the slow interactive flow would have caught via clarifying questions.
